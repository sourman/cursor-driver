# Bug: Enabling a custom OpenAI API key breaks Cursor's own models (composer-2.5 / grok) with "This model does not support custom API keys"

## Summary

When **"Use my own OpenAI API key"** is enabled (global `useOpenAIKey` flag),
Cursor's own proprietary models — **Composer 2.5 / 2.5 Fast / 2** (backend name
`grok-composer-2.5` / `grok-composer-2`) — stop working entirely. Any request to
them fails with:

> **This model does not support custom API keys.** Request ID: `94960f91-8b6a-467b-9f84-c248cde5f168`

This happens even though the custom key/endpoint is intended only for other
models (e.g. gpt-4o). The root cause is a **client-side routing function that
force-classifies every non-Claude, non-Gemini model as "use the custom OpenAI
key,"** including Cursor's own models. The request then reaches Cursor's backend
with a custom key attached, and the backend (correctly) rejects it.

## Environment

- Cursor **3.9.16** (Chrome 144 / Electron 40.10.3), Linux x64
- `workbench.glass.main.js` (Glass renderer)
- BYOK config live: `useOpenAIKey = true`, `openAIBaseUrl = https://router.loom.li`

## Reproduce

1. Settings → Models → enable **"Use my own OpenAI API key"**, enter any key + base URL.
2. Select **Composer 2.5** (or 2.5 Fast / 2) in chat/agent.
3. Send any message.

**Actual:** error toast — *"This model does not support custom API keys. Request ID: …"*.
**Expected:** Composer 2.5 routes to Cursor's own backend (it is a Cursor model
that no custom endpoint can serve), regardless of the custom-key setting.

Disabling "use my own key" immediately restores Composer 2.5.

## Root cause

Routing is decided in one pure function in `workbench.glass.main.js` (~line 12006).
Anchored by function name since the bundle is minified:

```js
function SJk(n){ return n.startsWith("claude-"); }                 // claude-key model?
function kJk(n){ return n.startsWith("gemini-"); }                 // google-key model?
function fHy(n, e){                                                 // e = applicationUserPersistentStorage
  return SJk(n) ? (e.useClaudeKey ? "anthropic" : undefined)
       : kJk(n) ? (e.useGoogleKey ? "google" : undefined)
       :           e.useOpenAIKey  ? "openai"   : undefined;        // ← DEFAULT branch
}
function I1h(n, e){ return fHy(n, e) !== undefined; }              // "BYOK-routed?"
```

The **default branch** assumes: *if a model isn't `claude-` or `gemini-` prefixed,
it's an OpenAI model the user's custom endpoint can serve.* That assumption is
true for `gpt-*` / `o3` / `o4-mini`, but **false for Cursor's own models** —
`composer-2.5` (→ `grok-composer-2.5`), `composer-2` (→ `grok-composer-2`), and
the `grok-*` family match neither prefix. They fall into the default branch and
get classified `"openai"` whenever the global `useOpenAIKey` is on.

`getApiKeyForModel` then attaches the custom key for them:

```js
getApiKeyForModel(n){
  const e = fHy(n, this._reactiveStorageService.applicationUserPersistentStorage);
  switch (e){
    case "anthropic": return this.claudeKey() ?? undefined;
    case "google":    return this.googleKey() ?? undefined;
    case "openai":    return this.openAIKey() ?? undefined;   // ← composer-2.5 lands here
    case undefined:   return;                                  // ← would correctly use Cursor's backend
  }
}
```

The request reaches Cursor's backend carrying the custom key for a
Cursor-proprietary model, which the backend rejects with
*"This model does not support custom API keys."* (That string is not present in
any local bundle — renderer or main — nor is it generated by the custom endpoint;
it is produced server-side, consistent with the attached Request ID.)

### Why `availableAPIKeyModels` doesn't save it

The list of models the custom endpoint actually serves (`availableAPIKeyModels`,
populated by probing `{openAIBaseUrl}/chat/completions`) is **only ever written,
never consulted by `fHy`/`I1h`/`getApiKeyForModel`.** So even when that list is
empty, or does not contain `grok-composer-2.5`, the misrouting still happens.
Routing is a pure function of the model-name prefix + the three global key flags —
nothing more.

### Per-model granularity is an illusion for OpenAI-family models

`getUseApiKeyForModel(n) === I1h(n, storage) === fHy(n, storage) !== undefined`,
which for any default-category model collapses to the **global** `useOpenAIKey`.
There is no per-model opt-out for OpenAI-family models, so enabling the custom key
for one model (gpt-4o) poisons every other default-category model — including
Cursor's own.

## Live evidence (captured from the running app over CDP)

The diagnostic tool `bin/cursor-byok-trace` recomputes `fHy` against the live
`applicationUserPersistentStorage` (via the existing `globalThis.__cursorRS` hook):

```
BYOK flags (live):
  useOpenAIKey  : True        openAIBaseUrl : https://router.loom.li
  useClaudeKey  : False       useGoogleKey  : False
  availableAPIKeyModels (what the endpoint serves): []

Per-model routing (fHy default branch):
  model              kind         routedVia        status
  composer-2.5       cursor-own   openai           BROKEN (backend rejects custom key)
  composer-2.5-fast  cursor-own   openai           BROKEN
  composer-2         cursor-own   openai           BROKEN
  grok-composer-2.5  cursor-own   openai           BROKEN
  gpt-4o / gpt-5 / o3 / o4-mini  openai  openai   BROKEN (endpoint doesn't serve them)
  glm-5.2            custom       openai           ok     (user's own model — served by endpoint)
  claude-sonnet-4    claude       (cursor backend) ok
  gemini-2.5-pro     google       (cursor backend) ok
```

Two things to note: `availableAPIKeyModels` is empty, yet `composer-2.5` is still
routed `"openai"`; and the user's *own* custom model (`glm-5.2`) routes to the
endpoint correctly — so the failure is specific to Cursor-served models caught by
the default branch, not to BYOK overall.

## Network capture (definitive)

tshark TLS-SNI capture on the Cursor host during a live repro (BYOK on, Composer
2.5 selected, message sent → error "This model does not support custom API keys",
Request ID `e6b36bdc-ed03-4f88-bbba-82e9afc49c1d`):

| Destination host | TLS handshakes |
|---|---|
| `api2.cursor.sh` (Cursor's backend) | **50** |
| `router.loom.li` (the configured custom endpoint) | **0** |

The failing Composer 2.5 request is delivered to **Cursor's own backend**, never
to the custom endpoint — yet it carries the custom key (because `fHy` flagged it
`"openai"`), so the backend rejects it. The custom endpoint is not involved at
all. This rules out any "the custom endpoint is rejecting the model" theory: the
entire failure is client-side misclassification (`fHy`) → server-side enforcement.


## Suggested fix (any one suffices)

1. **Exclude Cursor's own models from the default branch** — treat
   `composer-*` / `grok-*` (any Cursor-proprietary slug) as always-Cursor-backend
   in `fHy`, e.g. return `undefined` for them before the `useOpenAIKey` fallthrough.
2. **Consult `availableAPIKeyModels`** — only route a model via `"openai"` if the
   probed endpoint actually advertises it; otherwise fall through to Cursor's backend.
3. **Make the OpenAI key truly per-model** — `getUseApiKeyForModel` should reflect
   an explicit per-model toggle rather than collapsing to the global flag for the
   entire default category.

Option 1 is the smallest, most targeted change and cannot regress real BYOK
models (gpt-*/o3), which remain in the default branch.

### Verified locally

We confirmed Option 1 works. A one-line patch to `fHy`'s default branch —

```js
// before:  e.useOpenAIKey ? "openai" : undefined
// after:   e.useOpenAIKey && !/^(composer-|grok-)/.test(n) ? "openai" : undefined
```

— restores Composer 2.5 / grok (now routed to the Cursor backend, no custom key
attached) **while leaving BYOK intact for the user's own custom-endpoint models**
(verified with an OpenAI-compatible endpoint serving a non-OpenAI model: Composer
2.5 and the custom model both work simultaneously with `useOpenAIKey` on, no
toggling). Real BYOK models (gpt-*/o3) are untouched. This validates that the
default branch, not the BYOK plumbing, is the sole source of the failure.


## Impact

Any user who configures a custom OpenAI key — a supported, advertised feature —
loses access to Cursor's flagship Composer models with a misleading error that
blames the model rather than the key setting. The models work perfectly the moment
the custom key is disabled, which makes the failure feel arbitrary.

The impact is broader than Composer alone. Because the default branch catches
**every** non-`claude-`/non-`gemini-` model, it also hijacks the OpenAI-family
models Cursor proxies on subscription (`gpt-5`, `o3`, `o4-mini`) toward the custom
endpoint. Users whose custom endpoint is an OpenAI-*compatible* router serving
non-OpenAI models (e.g. `glm-5.2`) hit a particularly confusing failure pattern:
their own endpoint's models work, while every Cursor-served model in the default
category (Composer + the OpenAI family) silently breaks. The bug is invisible at
the UI level — there is no per-model indicator that a model has been force-tagged
for the custom key.

## Repro artifacts

- Diagnostic: `bin/cursor-byok-trace` (dumps live per-model routing) /
  `bin/cursor-byok-trace --watch` (logs renderer traffic to the custom endpoint).
- Network proof: `sudo tshark -i any -f 'tcp port 443' -Y 'tls.handshake.type==1'
  -T fields -e frame.time_relative -e tls.handshake.extensions_server_name` during
  a repro — shows all traffic going to `api2.cursor.sh`, none to the custom endpoint.
- Backend-name mapping (confirms composer is a Cursor/grok model, not OpenAI):
  `case "composer-2.5": return ["grok-composer-2.5"];` (in `TJk`, ~L12006).
- Local workaround (implements Suggested fix #1): `bin/cursor-byok-fix` patches
  `fHy` so `composer-*`/`grok-*` skip the custom key — lets Composer and custom
  models coexist with BYOK on.
